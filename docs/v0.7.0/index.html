Where is the AI coming from?

The setup path depends on whether you're bringing your own AI model, or connecting an agent that already has one.

🧑‍💻

I want to give tasks in plain English

Configure an API key. clawd-cursor runs its own 6-layer AI agent, reasons about your task, and executes it end-to-end. You type what you want — it figures out how.

clawdcursor doctor clawdcursor start REST /task
🤖

I'm connecting an agent that already reasons

No API key needed inside clawd-cursor. Your agent (Claude Code, Cursor, Windsurf, or your own code) calls the 33 tools directly. clawd-cursor is the body — your agent is the brain.

clawdcursor mcp REST /execute/{tool} Claude Code Cursor Windsurf
Transport modes

Three ways to connect

Same 33 tools, same desktop access. Different entry points depending on who's driving.

🔌
For agents

MCP stdio

Native MCP server for Claude Code, Cursor, Windsurf, and Zed. Your AI client discovers all 33 tools automatically — no HTTP plumbing needed.

# 1. One-time consent (required first) clawdcursor consent # 2. Add to Claude Code (~/.claude.json) { "mcpServers": { "clawd-cursor": { "command": "clawdcursor", "args": ["mcp"] } } }
⚠ Run clawdcursor consent first. MCP mode won't start without it.
🌐
For developers / custom agents

REST API

HTTP tool calls from any code. Start the server, then call individual tools or send full tasks. Works with any HTTP client.

# Start the server clawdcursor start # Call a raw tool POST /execute/desktop_screenshot POST /execute/mouse_click { "x": 450, "y": 300 } # OR send a plain-English task POST /task { "task": "open Chrome" } # List all tools + schemas GET /tools
🧠
For humans with an API key

CLI Agent

clawd-cursor is the agent. Configure your AI provider — it reasons, plans, and executes tasks end-to-end using the 6-layer pipeline.

# Auto-detect your AI providers clawdcursor doctor # Start the agent clawdcursor start # Send a task clawdcursor task "Open Gmail and send a draft to team@co.com" # Or interactive task console clawdcursor task
Safety

One-time consent

clawd-cursor gives AI models full desktop access. You must explicitly authorize this before any mode will start.

⚠️

Desktop Control Warning

This tool grants AI models: mouse + keyboard control, screenshot capture, clipboard access, app switching, browser DOM via CDP, and accessibility tree reads. Server binds to localhost only. Dangerous key combos are blocked.

Interactive clawdcursor consent
Non-interactive clawdcursor consent --accept
Check status clawdcursor consent --status
Revoke clawdcursor consent --revoke
Architecture

6-layer pipeline

Each layer tries to handle the task before escalating to the next. Cheaper layers run first — vision is the last resort, not the first call.

L0

Browser · Playwright

Handles browser-specific tasks via Playwright automation

text model
↓ falls through if not handled
L1

Action Router

Pattern-matched tasks: open apps, type text, shortcuts, navigate URLs — zero LLM calls

$0
↓
L1.5

Deterministic Flows

Hardcoded step-by-step scripts for known task patterns with ground-truth verification

$0
↓
L2

A11y + CDP Reasoner

Reads accessibility tree and Chrome DevTools Protocol. Text model reasons about what to click. Bounds-based click resolver — no vision needed

text model
↓ if element not found in a11y tree
L2.5

Vision Coordinate Spotter

Asks vision model "where is [element]?" Returns coordinates. Cheaper than full computer use

vision (coords only)
↓ last resort
L3

Computer Use

Full vision model with screenshot + action loop. Handles anything — native apps, legacy software, anything on screen

vision model

In MCP / REST mode, there is no pipeline — your agent calls tools directly. The pipeline only runs in CLI agent mode (clawdcursor start).

v0.6.3 → v0.7.0

What changed

Not a feature release — a reliability and openness overhaul.

Capability v0.6.3 v0.7.0
Architecture 4-layer pipeline 6-layer pipeline
Transport REST only REST + MCP stdio + tools-only server
Tools Monolithic — no tool exposure 33 discrete tools, OpenAI format
Success verification LLM self-reports (often wrong) Ground-truth action verifier
Loop protection Agent can repeat failed action forever No-progress detector + premature-done blocker
Click resolution Vision model guesses coordinates A11y bounds-based — zero LLM cost
Browser interaction Playwright only CDP — click by selector, read DOM
AI model support Anthropic-favored Truly model-agnostic, mixed providers
Platform dependency Tied to OpenClaw Fully standalone, auto-migrates data
Onboarding None — starts immediately First-run consent flow
Task logging Console output only Structured JSONL — verified vs unverified
